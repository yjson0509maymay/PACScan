# PACScan 설계 의사결정 기록

## ADR-001 — Streamlit을 프로토타입 UI로 사용

### 상태

채택

### 결정

의료진용 분석 화면과 시연 서비스는 Streamlit으로 구현한다.

### 이유

- Python 의료영상 처리 코드와 연결이 간단하다.
- 파일 업로드, 상태 표시 및 결과 시각화를 빠르게 구현할 수 있다.
- 모델이 완성되기 전에도 UI와 사용자 흐름을 검증할 수 있다.

### 한계

- Community Cloud는 FSL과 같은 대형 네이티브 의료영상 도구 실행에 적합하지 않다.
- 실제 서비스에서는 전처리 API 서버 또는 사전 생성된 결과 연결이 필요하다.

## ADR-002 — 실행은 원버튼, 결과 탐색은 탭

### 상태

채택

### 결정

사용자가 누르는 처리 버튼은 `분석 시작` 하나로 유지한다. 분석 완료 후 원본, 전처리, AI 분석 및 보고서는 탭으로 자유롭게 전환한다.

### 이유

- 단계별 반복 클릭을 줄인다.
- 처리 순서 오류를 예방한다.
- 의료진은 완료된 결과 사이를 자유롭게 비교할 수 있다.

## ADR-003 — N4 bias-field correction 제외

### 상태

채택

### 결정

`ref21order_v1` 시연 및 최종 전처리 설명에서 N4 보정을 사용하지 않는다.

### 이유

실제 데이터 비교 결과 N4를 적용하지 않은 방법이 더 적절한 결과를 보였다.

### 최종 표시 파이프라인

```text
DICOM → NIfTI → BET → 정합 → Min-Max 정규화 → 56×56×56 리사이즈
```

## ADR-004 — 실제 ref21order_v1 산출물을 시연에 사용

### 상태

채택

### 결정

시연 영상에서는 사전에 실제로 처리된 `ref21order_v1` 단계별 산출물을 사용한다.

### 이유

- BET 및 정합을 실시간 실행할 때 발생하는 대기시간을 제거한다.
- 시연 중 외부 실행파일 오류 위험을 낮춘다.
- 가짜 이미지가 아니라 실제 파이프라인 결과를 보여준다.

### 공개 시 주의

보고서와 발표에서는 `실제 전처리 산출물을 캐시해 시연`한다고 명시한다.

## ADR-005 — AI 결과는 모델 완료 전까지 데모로 표시

### 상태

임시 채택

### 결정

모델 체크포인트가 준비되기 전까지 확률, 히트맵 및 판독 문장에 `DEMO` 표시를 유지한다.

### 완료 조건

- 고정 모델 체크포인트 확보
- 단일 환자 추론 함수 확보
- 입력 shape 및 정규화 검증
- M3d-CAM 출력 검증
- 테스트 환자에 대한 재현성 확인

## ADR-006 — 의료 데이터는 공개 GitHub에 저장하지 않음

### 상태

채택

### 결정

실제 DICOM, NIfTI, 환자정보 및 원본 로그를 PACScan GitHub에 커밋하지 않는다.

### 허용

- 완전한 합성 예시 데이터
- 개인정보가 없는 UI 이미지
- 집계된 결과와 비식별 로그
