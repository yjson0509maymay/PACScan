# PACScan 분석 실행 로그 명세

개발 기록과 별도로 실제 분석 작업마다 구조화 로그를 생성한다.

## 권장 저장 구조

```text
run_id
├─ run.json
├─ steps.jsonl
├─ artifacts.json
└─ error.json (실패한 경우)
```

## run.json 필드

| 필드 | 설명 | 예시 |
|---|---|---|
| `run_id` | 분석 작업 고유 ID | `20260722_103015_a8f21c` |
| `started_at` | 시작 시각 | ISO 8601 |
| `finished_at` | 종료 시각 | ISO 8601 |
| `patient_hash` | 비식별 환자 해시 | SHA-256 일부 |
| `input_type` | 입력 유형 | `dicom_directory` |
| `dicom_count` | DICOM 파일 수 | 32 |
| `series_count` | 발견 시리즈 수 | 1 |
| `selected_series` | 선택된 T2 설명 | `T2 AXIAL` |
| `pipeline` | 파이프라인 이름 | `ref21order_v1` |
| `pipeline_commit` | 코드 버전 | Git SHA |
| `bet_frac` | BET 파라미터 | 0.5 |
| `normalization` | 정규화 | `minmax` |
| `bias_correction` | N4 사용 여부 | false |
| `target_shape` | 최종 shape | `[56,56,56]` |
| `model_mode` | 모델 상태 | `demo` 또는 `production` |
| `status` | 최종 상태 | `success`, `failed`, `cancelled` |
| `elapsed_sec` | 전체 시간 | 23.8 |

## 단계 로그

각 줄을 독립 JSON으로 기록한다.

```json
{"run_id":"...","step":"dicom_to_nifti","status":"success","elapsed_sec":1.4}
{"run_id":"...","step":"bet","status":"success","elapsed_sec":8.2}
{"run_id":"...","step":"registration","status":"success","elapsed_sec":12.7}
{"run_id":"...","step":"normalization","status":"success","elapsed_sec":0.4}
{"run_id":"...","step":"resize","status":"success","elapsed_sec":0.2}
```

## 산출물 로그

실제 파일 자체가 아니라 경로, shape, 크기 및 checksum을 기록한다.

```json
{
  "artifact_type": "final_nifti",
  "filename": "subject_preprocessed_56.nii.gz",
  "shape": [56, 56, 56],
  "size_bytes": 166564,
  "sha256": "..."
}
```

## 모델 결과

모델이 연결되기 전에는 다음처럼 기록한다.

```json
{
  "model_mode": "demo",
  "model_version": "not_connected",
  "probabilities_are_clinical": false
}
```

모델 연결 후에는 체크포인트 checksum, 입력 shape, 예측 확률, XAI 방법 및 추론시간을 추가한다.

## 저장 정책

- 로컬 시연: SQLite 또는 JSONL
- 배포 서비스: PostgreSQL 등 외부 영속 데이터베이스
- Streamlit Community Cloud 로컬 디스크를 영구 저장소로 사용하지 않는다.
- 오류 로그에 DICOM PatientName, PatientBirthDate 및 원본 PatientID가 포함되지 않도록 정제한다.
